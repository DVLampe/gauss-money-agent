# Gauss Money — AI-агент для churn & revenue отчёта

AI-агент генерирует синтетические данные по подписочной модели, вычисляет
ежемесячные KPI, проверяет качество данных и формирует бизнес-отчёт —
всё под управлением GPT-4o через механизм function calling.

---

## Быстрый старт

### Вариант 1 — веб-интерфейс (рекомендуется)

```bash
pip install -r requirements.txt
streamlit run app.py
```

Откроется браузер на `http://localhost:8501`. Введите OpenAI API Key, нажмите
**▶ Запустить агента** — прогресс отобразится пошагово, отчёт появится прямо
на странице, все файлы можно скачать кнопками.

### Вариант 2 — консоль

```bash
pip install -r requirements.txt
cp .env.example .env   # добавить OPENAI_API_KEY в .env
python agent.py
```

### Вариант 3 — онлайн, без установки зависимостей

Загрузите репозиторий на GitHub и подключите к
[Streamlit Community Cloud](https://streamlit.io/cloud) (бесплатно):
1. New app → выбрать репо → `app.py`
2. Settings → Secrets → добавить `OPENAI_API_KEY = "sk-..."`
3. Поделиться ссылкой с проверяющим — он открывает в браузере, ничего не устанавливает

|------|------------|

## Определения метрик

| Метрика | Определение |
|---------|-------------|
| `active_users` | Пользователи с `is_active=True` на конец месяца M |
| `paid_users` | Пользователи, у которых `payment_status='paid'` в месяце M |
| `churned_users` | Пользователи с `is_active=False` в месяце M (последняя запись) |
| `monthly_revenue` | `sum(amount_paid)` за месяц M |
| `churn_rate` | `churned_users[M] / active_users[M-1]`; для месяца 1 = 0 |
| `arpu` | `monthly_revenue[M] / active_users[M]` |

## Описание агентного подхода

### Роль агента

GPT-4o выступает **оркестратором**: получает одну высокоуровневую задачу
(«сформировать отчёт») и самостоятельно решает, какие инструменты вызвать,
в каком порядке и с какими аргументами. Он же является **guardrail** —
системный промпт явно запрещает вызывать `generate_report`, если хоть одна
проверка данных не прошла.

### Как устроен агент

Генерация данных — это **шаг подготовки**, отделённый от агента.
В продакшне здесь был бы реальный CSV из бизнес-системы. Агент получает
путь к файлу и работает только с аналитикой.

```
[Pre-step: подготовка данных]
  generate_data() → data/users.csv
  (в продакшне — реальный CSV из биллинговой системы)
         │
         │  передаём путь к файлу
         ▼
┌─────────────────────────────────────┐
│  GPT-4o (system prompt + история)   │  ← рассуждение и оркестрация
│  OpenAI function calling (tool use) │
└──────┬──────────────────────────────┘
       │  сообщения tool_call
       ▼
┌──────────────────────┐
│  Инструмент 1        │
│  calculate_metrics   │  ← KPI по каждому месяцу
│  (pandas)            │
└──────────┬───────────┘
           │
┌──────────▼───────────┐
│  Инструмент 2        │
│  validate_data       │  ← 8 детерминированных проверок
│  (pandas)            │
└──────────┬───────────┘
           │ (только если все ✓)
┌──────────▼───────────┐
│  Инструмент 3        │
│  generate_report     │  ← нарратив через GPT-4o sub-call
│  (GPT-4o sub-call)   │
└──────────────────────┘
```

Цикл работает до тех пор, пока GPT-4o не вернёт текстовый ответ без `tool_calls` —
тогда агент печатает финальное резюме и завершает работу.

### Где используются обычные расчёты кодом

- **Генерация данных** — NumPy: для каждого пользователя заранее определяется
  месяц оттока через rolling churn probability, что даёт воспроизводимую симуляцию.
- **Расчёт метрик** — pandas-агрегации; LLM не участвует.
- **Валидация** — восемь детерминированных pandas/numpy-проверок.

### Где используется LLM

1. **Внешний агент** (GPT-4o, `agent.py`): принимает решение о порядке вызовов
   инструментов и передаваемых аргументах.
2. **Внутренний sub-call** (GPT-4o, `tools/report_generator.py`): переводит
   числовую таблицу KPI в бизнес-язык: тренды, интерпретацию, рекомендации.
   `temperature=0.3` — для снижения галлюцинаций на числах.

### Какие промпты используются

**Системный промпт агента**
> «Ты — senior fintech data analyst AI agent. Тебе передан готовый CSV.
> Вызывай инструменты строго в порядке: 1 → calculate_metrics,
> 2 → validate_data, 3 → generate_report (только если все проверки прошли).»

**Промпт генерации отчёта**
Передаётся: полная KPI-таблица (markdown), сводка валидации, производные
статистики (total revenue, retention rate, месяц пикового/минимального оттока).
GPT-4o получает инструкцию написать структурированный отчёт из 6 разделов
с реальными числами, без общих фраз.

### Guardrails и ограничения

| Guardrail | Механизм |
|-----------|----------|
| Правильный порядок вызовов | Системный промпт задаёт точную последовательность |
| Нет отчёта на плохих данных | Промпт: вызывать `generate_report` только если все проверки ✓ |
| Воспроизводимость | Фиксированный `seed=42` в `generate_data` |
| Точность расчётов | 8 детерминированных проверок до того, как LLM увидит данные |
| Снижение риска галлюцинаций | `temperature=0.3`; промпт явно якорит LLM на реальных числах |
| Безопасность схемы | Каждый инструмент возвращает строго структурированный JSON; агент не видит сырые DataFrame |

### Почему такой подход

- **Минимум зависимостей**: прямой OpenAI API + pandas, без тяжёлых фреймворков.
- **Прозрачность**: каждый вызов инструмента и его результат виден в консоли.
- **Воспроизводимость**: фиксированный seed + детерминированные расчёты → одинаковый CSV при каждом запуске.
- **Разделение ответственности**: чистые Python-инструменты тестируются независимо от LLM.
- **LLM добавляет ценность там, где нужно**: интерпретация трендов и бизнес-язык, а не арифметика.

## Проверки качества данных

| № | Проверка | Что валидируется |
|---|----------|-----------------|
| 1 | Нет дублей `(user_id, month)` | Отсутствие дублирующихся записей |
| 2 | Консистентность выручки | `metrics.monthly_revenue == sum(amount_paid)` в сырых данных |
| 3 | Монотонное убывание active_users | Пользователи не реактивируются после оттока |
| 4 | Churn rate ∈ [0, 1] | Нет невозможных значений |
| 5 | Консистентность сумм платежей | `failed` → `amount_paid=0`; `paid` → `amount_paid=monthly_price` |
| 6 | Нет активности после оттока | `is_active=False` только в последней записи пользователя |
| 7 | Сохранение потока пользователей | `active[M] + churned[M] == active[M-1]` для всех M > 1 |
| 8 | Все пользователи начинают с месяца 1 | Ни один пользователь не появляется посередине когорты |
