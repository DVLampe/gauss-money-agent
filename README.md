# Gauss Money — AI-агент для churn & revenue отчёта

AI-агент генерирует синтетические данные по подписочной модели, вычисляет
ежемесячные KPI, проверяет качество данных и формирует бизнес-отчёт —
всё под управлением GPT-4o через механизм function calling.

---

## Определения метрик

| Метрика | Определение |
|---------|-------------|
| `active_users` | Пользователи с `is_active=True` на конец месяца M |
| `paid_users` | Пользователи, у которых `payment_status='paid'` в месяце M |
| `churned_users` | Пользователи с `is_active=False` в месяце M (последняя запись) |
| `monthly_revenue` | `sum(amount_paid)` за месяц M |
| `churn_rate` | `churned_users[M] / active_users[M-1]`; для месяца 1 = 0 |
| `arpu` | `monthly_revenue[M] / active_users[M]` |

## Описание агентного подхода

### Роль агента

GPT-4o участвует в двух ключевых шагах:
1. **Генерация отчёта** — переводит KPI-таблицу в бизнес-нарратив (`report_generator`).
2. **Ревью отчёта** — критически оценивает написанное: ясность языка, соответствие данным,
   конкретность рекомендаций (`report_reviewer`). Если находит проблемы — возвращает
   обратную связь и отчёт перегенерируется. Максимум 3 попытки.

### Как устроен агент

Генерация данных — это **шаг подготовки**, отделённый от агента.
В продакшне здесь был бы реальный CSV из бизнес-системы. Агент получает
путь к файлу и работает только с аналитикой.

```
[Pre-step: подготовка данных]
  generate_data() → data/users.csv
  (в продакшне — реальный CSV из биллинговой системы)
         │
         │  передаём путь к файлу
         ▼
┌─────────────────────────────┐
│  Шаг 1: calculate_metrics   │  ← KPI по каждому месяцу (pandas)
└──────────┬──────────────────┘
           │
┌──────────▼──────────────────┐
│  Шаг 2: validate_data       │  ← 8 проверок: HARD блокируют отчёт,
│  (pandas)                   │    SOFT — предупреждение, отчёт продолжается
└──────────┬──────────────────┘
           │ если нет HARD-ошибок
┌──────────▼──────────────────┐     ┌───────────────────────────────┐
│  Шаг 3: generate_report     │◄────│  feedback (если ревью не ОК)  │
│  (GPT-4o sub-call)          │     └───────────────────────────────┘
└──────────┬──────────────────┘
           │
┌──────────▼──────────────────┐
│  Шаг 4: review_report       │  ← GPT-4o проверяет отчёт:
│  (GPT-4o sub-call)          │    ясность → точность данных → рекомендации
└──────────┬──────────────────┘
           │ approved=true → готово
           │ approved=false → вернуться на Шаг 3 (макс. 3 попытки)
           ▼
      Финальное резюме агента
```

### Где используются обычные расчёты кодом

- **Генерация данных** — NumPy: для каждого пользователя заранее определяется
  месяц оттока через rolling churn probability, что даёт воспроизводимую симуляцию.
- **Расчёт метрик** — pandas-агрегации; LLM не участвует.
- **Валидация** — восемь детерминированных pandas/numpy-проверок (LLM не участвует).

### Где используется LLM

1. **Генерация отчёта** (GPT-4o, `tools/report_generator.py`): переводит
   числовую таблицу KPI в бизнес-язык: тренды, интерпретацию, рекомендации.
   При повторных попытках получает feedback от ревьюера и перерабатывает текст.
   `temperature=0.3` — для снижения галлюцинаций на числах.
2. **Ревью отчёта** (GPT-4o, `tools/report_reviewer.py`): независимая проверка
   по трём критериям — ясность языка для нетехнического читателя, соответствие
   числам из metrics.csv, конкретность рекомендаций. Возвращает
   `{"approved": bool, "issues": [...], "feedback": "..."}`. `temperature=0.2`.
3. **Итоговое резюме** (GPT-4o, `agent.py`): 2–3 предложения с ключевыми
   цифрами (revenue, retention, предупреждения).

### Какие промпты используются

**Промпт генерации отчёта** (`report_generator.py`)
Передаётся: полная KPI-таблица (markdown), сводка валидации, производные
статистики (total revenue, retention rate, месяц пикового/минимального оттока).
GPT-4o получает инструкцию написать структурированный отчёт из 6 разделов
с реальными числами, без общих фраз. При повторной попытке в промпт добавляется
раздел «IMPORTANT: Feedback from previous review» с конкретными замечаниями.

**Промпт ревью отчёта** (`report_reviewer.py`)
Передаётся: metrics.csv (ground truth) + текст отчёта.
GPT-4o-ревьюер проверяет три критерия:
- Ясность для нетехнического руководителя
- Точность всех чисел (сравнивает с source metrics)
- Конкретность и обоснованность 3 рекомендаций
Отвечает JSON: `{"approved": bool, "issues": [...], "feedback": "..."}`.
Если `approved=false` — feedback передаётся в следующую попытку генерации.

### Guardrails и ограничения

| Guardrail | Механизм |
|-----------|----------|
| Правильный порядок шагов | Жёстко задан в коде (sequential pipeline, не LLM-loop) |
| Нет отчёта на плохих данных | HARD-ошибки в validate_data → немедленная остановка; SOFT-предупреждения → отчёт генерируется с пометкой |
| Качество отчёта | review_report проверяет ясность, точность данных и рекомендации; при отказе — до 3 перегенераций с feedback |
| Воспроизводимость | Фиксированный `seed=42` в `generate_data` |
| Точность расчётов | 8 детерминированных проверок до того, как LLM увидит данные |
| Снижение риска галлюцинаций | `temperature=0.3` (генерация), `temperature=0.2` (ревью); промпты явно якорят LLM на реальных числах |
| Безопасность схемы | Каждый инструмент возвращает строго структурированный JSON |

### Почему такой подход

- **Минимум зависимостей**: прямой OpenAI API + pandas, без тяжёлых фреймворков.
- **Прозрачность**: каждый вызов инструмента и его результат виден в консоли.
- **Воспроизводимость**: фиксированный seed + детерминированные расчёты → одинаковый CSV при каждом запуске.
- **Разделение ответственности**: чистые Python-инструменты тестируются независимо от LLM.
- **LLM добавляет ценность там, где нужно**: интерпретация трендов и бизнес-язык, а не арифметика.

## Проверки качества данных

| № | Тип | Проверка | Что валидируется |
|---|-----|----------|------------------|
| 1 | **HARD** | Нет дублей `(user_id, month)` | Отсутствие дублирующихся записей |
| 2 | **HARD** | Консистентность выручки | `metrics.monthly_revenue == sum(amount_paid)` в сырых данных |
| 3 | SOFT | Монотонное убывание active_users | Предупреждение, если были реактивации / win-back |
| 4 | **HARD** | Churn rate ∈ [0, 1] | Нет невозможных значений |
| 5 | **HARD** | Консистентность сумм платежей | `failed` → `amount_paid=0`; `paid` → `amount_paid=monthly_price` |
| 6 | SOFT | Нет активности после оттока | Предупреждение, если были повторные подписки |
| 7 | **HARD** | Сохранение потока пользователей | `active[M] + churned[M] == active[M-1]` для всех M > 1 |
| 8 | **HARD** | Все пользователи начинают с месяца 1 | Ни один пользователь не появляется посередине когорты |

**HARD** — блокирует генерацию отчёта при провале.  
**SOFT** — выводит предупреждение; отчёт генерируется несмотря на провал (реактивации и win-back реальны в продакшне).
